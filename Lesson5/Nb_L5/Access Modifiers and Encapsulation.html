Implementation hiding¶

All good programmers know, that the software becomes good after rewriting, often for many times. When you write your code and come back to it later you might see the parts of code, that could be written in much better way. This is one of the prime motivations for refactoring , which rewrites working code to make it more readable, understandable, and thus maintainable.

There is something that you should know about when you want to improve your code. Sometimes you have consumers of your classes that rely on some of your code staying the same. On one hand you want to improve your code, from the other hand, consumers want things to be the same.

Thus a primary consideration in object-oriented design is to “separate the things that change from the things that stay the same.”

This can be achieved through convention. For example, the library programmer must agree not to remove existing methods when modifying a class in the library, since that would break the client programmer’s code.

But the story with the class fields is a little bit different. How can you know which fields of the class will be used? What if you want to remove some fields and your changes will break the clients code?

To solve this problem, Java provides access specifiers to allow the library creator to say what is available to the client programmer and what is not.

The levels of access is following: public, private, protected, package-private

First rule we should follow:

Make everything as private as possible.

Yes, that is. Start writing your class and give your class members the most restrictive access level (private). Only expose those methods or field which you want clients to use.

But how can you bundle related functionality in a cohesive library unit? This is controlled with the package keyword in Java.

Package¶

A package contains a group of classes, organized together under a single namespace
In [1]:
class PakageExample {
    public static void main(String[] args) {
        java.util.mypackage.one.ArrayList arrayList = new java.util.mypackage.one.ArrayList();
    }
}

For example, let's consider a standart utilities library of Java, organized under the namespace java.util.

This way of declaring a variable is tedious. So you can use import keyword instead.

In [3]:
import java.util.ArrayList;

public class FullPathExample {

    public static void main(String[] args) {
        mypackage.one.ArrayList list = new mypackage.one.ArrayList();
    }
}

Now you can use mypackage.one.ArrayList with no qualification. However, none of the other classes in java.util are available. To import everything, you use the * :
import java.util.*;

The reason for all this importing is to provide a mechanism to manage namespaces, because there is potential name's clashing of your classes. Namespaces can solve all our problems with names clashing.

Most of the examples you saw so far your was put in a single file and was designed for local use, so they haven’t bothered with package names. However, they were in a package implicitly - default package.

If you create a class only for your personal user - you can omit packages and write everything in the default package. But if you want your program to be friendly to other programs you need to consider using packages.

Now, if someone wants to use MyClass or, for that matter, any of the other public classes in hiding.mypackage, they must use the import keyword to make the name or names in hiding.mypackage available or use the full path to your class including the class name: hiding.mypackage.MyClass

IMPORTANT!

The package name should match the path to this file in your operation system. For example, if you declared a package name like edu.java.mypack, you should create a folder hierarchy on your computer. Like /Users/MyUser/testfolder/edu/java/mypack/MyClass.java

Always put the classes which belongs to the same package in the same directory which matches the package name.

Collisions¶

What happens if two libraries are imported via * and they include the same names?

file1

package com.mypackage;

public class mypackage.one.ArrayList {

}

file2


    import java.util.*;
    import com.mypackage.*;

    public class CollisionDemo {
        public static void main(String[] args) {
            mypackage.one.ArrayList myList = new mypackage.ArrayList(); // will be an error
        }
    }

Since you imported everything from java.util package and com.mypackage package two classes with the same name were imported. When you try to create an object of the mypackage.one.ArrayList class, the compiler doesn't know what object actually you want to create. So, it will complain and ask you to be explicit.

file2


    import java.util.*;
    import com.mypackage.*;

    public class CollisionDemo {
        public static void main(String[] args) {
            com.mypackage.one.ArrayList myList = new com.mypackage.ArrayList(); // It's OK
        }
    }

Java Access Specifiers¶

The Java access specifiers public, protected , and private are placed in front of your class members(fields and methods). Each access specifier only controls the access for that particular definition. If you don’t provide an access specifier, it means “package access.” So one way or another, everything has some kind of access control.

Package access¶

Default access has no keyword, and is commonly called package access (sometimes “friendly”). It means that all the other classes in the current package have access to that member.

To all the classes outside of this package, the member appears as private . Since a compilation unit—a file—can belong to only a single package, all the classes within a single compilation unit are automatically available to each other via package access. Package access groups related classes into a package so they can easily interact with each other.

Code from another package has only access to a member of a given class only if it is public. If you want to give access to your class member you can also create accessor/mutator also known as getter and setter.

Let's look at the example of package-private access in the project PackageAccessDemo.

Public access¶

When you use the public keyword, it means the member declaration that immediately follows public is available to everyone, in particular to the client programmer who uses the library.

There are no pitfalls when we are talking about public members. Example is in the project PublicAccessDemo.

Private access¶

The private keyword means no one can access that member except the class that contains that member, inside methods of that class. Other classes in the same package cannot access private members. With private, you can freely change that member without worrying whether it affects another class in the same package.

Here is some example of private members:

class Bank {
   private double amountOfMoney;
}

Protected access¶

The protected keyword deals with a concept called inheritance, that takes an existing class—which we refer to as the base class — and adds new members to that class without touching the existing class. You can also change the behavior of existing members of the class.
